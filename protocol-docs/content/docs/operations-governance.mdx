This page is written for protocol operators, deployers, integrators, and risk managers.

## Deployment Order

The factory/core pattern is intentionally two-step:

1. Deploy `RLDMarketFactory` with `CORE = address(0)`.
2. Deploy `RLDCore(factoryAddress, poolManager)`.
3. Call `RLDMarketFactory.initializeCore(coreAddress)` from the deployer.
4. Use `RLDMarketFactory.createMarket()` to deploy market-specific infrastructure.

`initializeCore()` can only be called once by the original deployer.

## Market Launch Checklist

| Check | Required action |
| --- | --- |
| Core initialized | `RLDMarketFactory.CORE()` must be non-zero. |
| Oracle wiring | `rateOracle`, `markOracle`, and optional `spotOracle` must match expected interfaces and quote directions. |
| Position token ownership | `PositionToken` mint/burn rights must be controlled by the intended protocol component. |
| Broker verifier | Market config must point to the correct `BrokerVerifier`. |
| Funding model | Standard markets use `StandardFundingModel`; CDS markets may use `CDSDecayFundingModel`. |
| Funding period | Choose a deliberate period, not an accidental default. |
| Debt cap | Set finite cap for early markets unless explicitly unlimited. |
| Liquidation params | Pack base discount, max discount, and slope correctly. |
| Deposit route | Register `BrokerRouter` deposit route for wrapped collateral flows. |
| Pool key | Verify collateral/position token ordering, fee, tick spacing, and hook. |
| Indexer | Confirm market, pool, events, candles, and account state are indexed. |
| Frontend config | Ensure infrastructure addresses populate `marketInfo`. |

## Risk Parameters

| Parameter | Meaning |
| --- | --- |
| `minColRatio` | Required ratio for new debt minting. |
| `maintenanceMargin` | Liquidation threshold for existing positions. |
| `liquidationCloseFactor` | Max close amount for non-underwater liquidations. |
| `fundingPeriod` | Timescale for standard mark-index funding. |
| `badDebtPeriod` | Timescale for socializing bad debt through NF. |
| `debtCap` | Max economic debt principal times NF. |
| `minLiquidation` | Dust protection for liquidations. |
| `liquidationParams` | Packed liquidation discount configuration. |
| `decayRateWad` | Annualized CDS decay parameter. |
| `brokerVerifier` | Valid broker source for this market. |

Curators propose risk updates through `RLDCore.proposeRiskUpdate()`. Updates are timelocked by `CONFIG_TIMELOCK`, currently 7 days.

## Parameter Change Process

1. Draft proposed parameter values with rationale and simulation evidence.
2. Confirm frontend and indexer can represent the new values.
3. Call `proposeRiskUpdate()`.
4. Monitor the pending update during the timelock.
5. Let the update apply when `_getEffectiveConfig()` is reached by an interaction.
6. Publish the applied change and expected user impact.

Avoid bundling unrelated risk changes. Margin, funding, liquidation, and debt caps affect different user groups and should be reviewable independently.

## Monitoring

Operators should monitor:

| Signal | Why it matters |
| --- | --- |
| Index price | Drives debt value and user PnL. |
| Mark price | Drives funding and execution quality. |
| Mark-index spread | Indicates funding pressure and arbitrage demand. |
| Normalization factor | Accumulates funding and bad debt socialization. |
| Total debt and debt cap utilization | Prevents overgrowth and detects usage spikes. |
| Broker health distribution | Identifies liquidation risk before cascades. |
| Bad debt | Signals failed liquidations or market stress. |
| Oracle freshness and bounds | Prevents stale or invalid pricing. |
| Pool TVL and depth | Determines slippage and liquidation resilience. |
| TWAMM order state | Critical for bonds and duration-managed strategies. |
| CDS withdrawal queues | Detects underwriter exit pressure. |
| Global settlement timestamp | Terminal market mode. |

## Incident Response

For severe oracle, funding, or settlement incidents:

1. Identify affected markets by `MarketId`.
2. Freeze UI actions that can worsen user state if needed.
3. Publish the observed invariant breach or data issue.
4. Confirm whether liquidations are valid under current oracle state.
5. If CDS-specific and configured, coordinate settlement-module actions.
6. Avoid emergency parameter changes that create hidden liquidation effects unless explicitly governed.

Core does not include a universal pause in the reviewed implementation, so operational controls depend on frontend gating, periphery controls, market parameters, or settlement modules.

## Production Readiness Gaps

The following items should be closed before a tier-1 mainnet launch:

| Area | Gap |
| --- | --- |
| CDS settlement | Implement, test, and audit the production settlement module. |
| Oracle assurance | Add or integrate TWAR, utilization/IRM validation, and stale-data handling. |
| Frontend validation | Centralize validation for margin, slippage, duration, debt cap, and CDS state. |
| Tests | Expand full-stack `RLDCore` + broker + periphery tests beyond DEX-heavy coverage. |
| Monitoring | Add dashboards for NF, bad debt, health, queue state, and mark-index spreads. |
| Docs | Keep contract addresses, market params, and risk disclosures versioned per deployment. |

## Source Of Truth

For protocol accounting, contracts are the source of truth. The frontend, paper, simulations, and this documentation explain the model, but user balances, debt, settlement state, and liquidation eligibility are determined by deployed contracts and oracle outputs.
