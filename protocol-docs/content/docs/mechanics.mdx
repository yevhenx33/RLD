title: Market Mechanics
description: The lifecycle of an RLD market from deployment to minting, trading, funding, repayment, and settlement.
---

This page describes the canonical lifecycle of a standard RLD market. CDS markets reuse much of the same machinery but add withdrawal delay, decay funding, and terminal settlement.

## Market Deployment

`RLDMarketFactory.createMarket()` performs the market bootstrap:

1. Validate deploy parameters and non-zero dependencies.
2. Precompute the deterministic `MarketId`.
3. Deploy a `PrimeBrokerFactory`.
4. Deploy a `BrokerVerifier`.
5. Clone and initialize the `PositionToken` for `wRLP`.
6. Initialize the v4/Ghost pool at the current oracle index price.
7. Register the market in `RLDCore`.
8. Verify the returned `MarketId`.

The canonical market key is:

```text
MarketId = keccak256(collateralToken, underlyingToken, underlyingPool)
```

This allows separate markets for the same asset if the underlying pool differs.

## Minting A Short

A short is debt creation:

```text
collateral in broker
  -> broker calls RLDCore.lock()
  -> broker receives lockAcquired()
  -> broker calls RLDCore.modifyPosition(deltaDebt > 0)
  -> core mints wRLP to broker
  -> broker/router may sell wRLP into the pool
  -> core checks broker solvency before lock exits
```

Core ignores `deltaCollateral` for accounting because collateral is held by the broker. The parameter remains in the interface for compatibility, but the broker's NAV is the solvency source.

Opening new debt uses `minColRatio`, the stricter initial margin threshold.

## Buying A Long

A long is a token purchase:

1. User funds the broker with collateral.
2. `BrokerRouter.executeLong()` swaps collateral into `wRLP`.
3. The broker holds `wRLP` as an asset.

The long benefits from rising rates because the index value of `wRLP` rises with `P_index = 100 * r_t`. If the long is held inside a broker, its value contributes to NAV.

## Closing A Short

A short closes by burning `wRLP` against principal:

```text
broker buys or already holds wRLP
  -> broker calls modifyPosition(deltaDebt < 0)
  -> PositionToken.burn(broker, amount)
  -> debtPrincipal decreases
  -> collateral becomes withdrawable if account remains solvent
```

The periphery path is `BrokerRouter.closeShort()`, which swaps collateral to `wRLP` and repays debt.

## Closing A Long

A long closes by swapping `wRLP` back to collateral:

```text
BrokerRouter.closeLong(broker, amountIn, poolKey)
```

The router validates pool keys and returns proceeds to the broker.

## Funding Application

Funding is lazy. `RLDCore._applyFunding()` runs during `modifyPosition()` and `liquidate()`, and there is a testnet wrapper `applyFunding()`.

The sequence is:

1. Read `oldNormFactor` and `lastUpdateTimestamp`.
2. Ask the market funding model for `newNormFactor`.
3. Update market state if the factor changed.
4. If bad debt exists, bleed a chunk into NF.
5. Update `lastUpdateTimestamp`.

Funding affects all debtors through one scalar. No per-account funding transfers are needed.

## Solvency

Core computes debt value:

```text
trueDebt = debtPrincipal * normalizationFactor
debtValue = trueDebt * indexPrice
```

Then it queries the broker:

```text
totalAssets = broker.getNetAccountValue()
netWorth = totalAssets - debtValue
requiredMargin = debtValue * (ratio - 1)
solvent = netWorth >= requiredMargin
```

If `getNetAccountValue()` reverts, core treats the broker as insolvent for liquidation purposes.

## Debt Cap

Debt caps are enforced in economic terms after minting updates token supply:

```text
trueTotalDebt = PositionToken.totalSupply() * normalizationFactor
trueTotalDebt <= debtCap
```

The cap is skipped when set to `type(uint128).max`.

## Bad Debt

If liquidation cannot recover enough value, the leftover principal becomes market bad debt:

```text
state.badDebt += uncoveredPrincipal
position.debtPrincipal = 0
```

Bad debt is socialized gradually by increasing `normalizationFactor` over `badDebtPeriod`. If total `wRLP` supply is zero, bad debt remains frozen until new supply exists.

## Global Settlement

CDS-capable markets can configure a non-zero `settlementModule`. That module can call:

| Function | Effect |
| --- | --- |
| `enterGlobalSettlement()` | Sets `globalSettlementTimestamp` and blocks future `modifyPosition()` for the market. |
| `invalidateBrokerWithdrawalQueue()` | Invalidates a broker withdrawal queue during settlement sweeps. |

The current core exposes the hooks, timestamp, withdrawal invalidation, and halt condition. Production settlement policy lives in the configured settlement module.
