RLD's safety model is margin-based. Every debt-bearing broker must maintain enough net account value to support its scaled debt at the current index price.

## Solvency Formula

Core computes:

```text
trueDebt = debtPrincipal * normalizationFactor
debtValue = trueDebt * indexPrice
netWorth = brokerAssets - debtValue
requiredMargin = debtValue * (ratio - 1)
solvent = netWorth >= requiredMargin
```

The ratio is:

| Operation | Ratio |
| --- | --- |
| New debt minting | `minColRatio` |
| Existing position maintenance | `maintenanceMargin` |
| Liquidation eligibility | `maintenanceMargin` |

This distinction lets new borrowing require stricter margin while existing accounts can be liquidated at the lower maintenance threshold.

## What Can Break Solvency

| Cause | Effect |
| --- | --- |
| Borrow rate rises | `P_index` rises, increasing debt value for shorts. |
| NF rises | True debt increases across the market. |
| Collateral price falls | Broker NAV decreases, especially in CDS markets with volatile collateral. |
| TWAMM or LP assets lose value | Broker strategy assets contribute less to NAV. |
| Bad debt socialization | NF increases as bad debt is spread across remaining supply. |

## Liquidation Eligibility

Anyone can call:

```text
RLDCore.liquidate(marketId, broker, debtToCover, minCollateralOut)
```

The target must be a valid broker from the configured broker verifier and below maintenance margin. Non-broker targets revert as invalid.

## Liquidation Flow

```text
liquidator
  -> RLDCore.liquidate()
     -> apply funding
     -> verify broker and insolvency
     -> enforce min liquidation and close factor
     -> cache index price and broker NAV
     -> reduce debt optimistically
     -> ask liquidation module for seize amount
     -> call PrimeBroker.seize()
     -> burn wRLP from broker and liquidator
     -> register bad debt if needed
     -> enforce minCollateralOut
```

The liquidator receives collateral. `wRLP` already inside the broker is burned first to reduce the amount the liquidator must supply.

## Close Factor

The close factor is dynamic:

| Broker state | Max liquidation |
| --- | --- |
| Assets cover debt value but margin is insufficient | Configured `liquidationCloseFactor`. |
| Assets are below debt value | Up to 100% of debt. |

This protects users from full liquidation when partial liquidation can restore health, while letting the protocol fully close underwater positions.

## Dutch Liquidation Bonus

`DutchLiquidationModule` calculates a health-based bonus:

```text
healthScore = collateralValue / (debtValue * maintenanceMargin)
bonus = baseDiscount + (1 - healthScore) * slope
bonus = min(bonus, maxDiscount)
```

The seize amount is:

```text
seizeAmount = debtToCover * NF * indexPrice * (1 + bonus) / spotPrice
```

The worse the health score, the larger the liquidation incentive, capped by `maxDiscount`.

## Seize Priority

`PrimeBroker.seize()` is designed to reduce market impact:

1. Use liquid `wRLP` and collateral already in the broker.
2. Cancel or claim TWAMM order if needed.
3. Remove V4 LP liquidity if needed.
4. Send extracted `wRLP` to core for burn.
5. Send collateral to the liquidator.

This avoids forcing every liquidator to buy the full debt amount in the market before liquidation.

## Bad Debt

If seized assets are insufficient and debt remains, core registers bad debt:

```text
state.badDebt += remainingPrincipal
position.debtPrincipal = 0
```

Bad debt is socialized over time by increasing NF:

```text
chunk = badDebt * dt / badDebtPeriod
NF += chunk / totalSupply
badDebt -= chunk
```

The chunk has a minimum based on total supply to avoid infinitely slow socialization.

## Broker NAV Limits

V1 brokers can track:

| Tracked item | Limit |
| --- | --- |
| Active V4 LP position | One position NFT. |
| Active TWAMM order | One order. |

Untracked positions are invisible to `getNetAccountValue()` and therefore do not support solvency. This keeps gas bounded but requires frontend and operators to track the right assets.

## Risk Register

| Risk | Mitigation in code or design | Remaining concern |
| --- | --- | --- |
| Oracle manipulation | Rate caps, Ghost observations, proposed TWAR/IRM checks. | Production oracle hardening must be completed and audited. |
| Broker NAV revert | Core treats reverts as insolvent. | Valid brokers must avoid accidental NAV reverts. |
| Correlated CDS collateral | CDS design calls for ETH/stETH or uncorrelated collateral. | Market configuration must enforce this. |
| Funding shock | Exponential NF update and configurable period. | Aggressive parameters can liquidate users. |
| Liquidation cascades | Broker burns internal `wRLP` first and unwinds assets only as needed. | Thin liquidity can still create mark moves. |
| Bad debt | Gradual NF socialization. | If supply is zero, bad debt is frozen. |
| Governance parameter abuse | Curator timelock for risk updates. | Oracle/governance attack assumptions still require policy. |

## Liquidator Checklist

1. Monitor `isSolvent()` or indexer health values.
2. Simulate `liquidate()` with current funding included.
3. Estimate broker `wRLP` and collateral extraction.
4. Set `minCollateralOut`.
5. Hold enough `wRLP` for any amount not extracted from the broker.
6. Account for gas, slippage, and possible TWAMM/LP unwinds.
